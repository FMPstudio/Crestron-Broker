# CORE BUSINESS LOGIC

There are 4 logical inputs:
- input 1
- input 2
- input 3
- input 4

Each broker command tells the broker:
“Route device X to input Y”.

Example:
- command: 1,7
  means:
  route device ID 07 to input 1

The broker must maintain state so it knows which device is currently assigned to each input.

If the broker receives:
- 1,7

then it must:
1. determine from cache/current state which device is currently routed to input 1
2. on the currently assigned device for input 1:
   - apply payload: Multicast Video Stream Disable.json
   - apply payload: Multicast Audio Stream Disable.json
3. on the new target device ID=07:
   - apply payload: Multicast video input 1.json
   - apply payload: Multicast audio input 1.json
   - apply payload: Multicast Video Stream Enable.json
   - apply payload: Multicast Audio Stream Enable.json
4. update the cache/state
5. send this acknowledgement back over WebSocket:
   - 1,7 OK!

Another example:
- command: 1,6

then:
1. look up which device is currently on input 1
2. disable video/audio stream on the old device
3. configure device ID=06 to input 1 using the matching payload files
4. enable video/audio stream on device ID=06
5. update state
6. respond:
   - 1,6 OK!

Another example:
- command: 3,7

then:
1. look up which device is currently assigned to input 3
2. disable video/audio stream on that old device
3. on device ID=07:
   - apply Multicast video input 3.json
   - apply Multicast audio input 3.json
   - apply Multicast Video Stream Enable.json
   - apply Multicast Audio Stream Enable.json
4. update state
5. respond:
   - 3,7 OK!

# STARTUP / STATE RECONSTRUCTION

Critical requirement:
The broker must not trust only its previous cache.

On startup it must:
1. connect/login to all known devices
2. query their current stream configuration/state via GET
3. reconstruct which device is currently routed to which input based on multicast addresses and/or payload mapping
4. write a local cache/state file

Reason:
- broker may restart
- devices may have been changed manually outside the broker
- cache may be stale

You must implement a robust bootstrap/sync phase.
